데이터 맵 구성 · WHAT TO COLLECT, ANALYZE & DERIVE
이번에는 데이터 맵을 구성해보도록 하겠습니다. 데이터 맵은 데이터 분석 및 서비스 기획을 수행함에 있어, 어떤 데이터를 수집할 지, 어떤 데이터로 분석할지, 파생변수는 어떻게 만들지를 결정하는 단계입니다. 실무에서는 분석을 수행할 데이터가 실제로 있다고 하더라도, 이 데이터 맵을 정리하는 작업을 수행합니다. 프로젝트의 기간에 따라 프로젝트 참여 인원이 달라질 수 있고, 인수인계를 하거나, 서비스 기획의 경우, 개발자가 해당 분석 내용을 확인할 때, 어떤 데이터로 분석을 수행했는지 명확히 작성되어 있어야 합니다. 아래 항목들을 확인하면서 데이터맵을 작성해보세요.

먼저 기업에서 데이터를 구성하기 전, 어떤 데이터를 수집/구성해야 할 지 결정해야 합니다. 기업에서는 데이터가 없는 경우, 또는 외부 데이터를 추가로 수집하는 경우, “특성 요인도 (Fish Bone Chart)”라는 표를 작성해 데이터를 수집합니다. 특성 요인도는 문제의 원인을 시각적으로 분석하여 이해하고, 그 원인을 이해하고 해결책을 찾는 기법입니다. 제조/공정에서 사용하기 시작해, 현재는 마케팅이나 CRM과 같은 다양한 분야에서 데이터를 수집하고 정의하는 단계에서 사용합니다. 어골도(생선의 뼈 모양)라고 불리는 특성 요인도는 데이터 구조화의 마인드 맵과 같습니다. 생선 머리엔 우리가 달성하고자 하는 목적(목표)를 작성하고, 큰 줄기에 각각의 핵심 요소들을 작성하여 가지를 뻗어나가는 형태로 작성합니다. 아래 특성 요인도의 예시를 보면 더 이해하기 쉽습니다.

빨간색 네모난 부분엔 비즈니스 시나리오를 기반으로 설정한 목적(또는 목표)가 들어갑니다. 그리고 큰 생선 뼈 줄기를 따라, 크게 나눠진 줄기(파란색 네모난 부분)엔 해당 목표를 달성하기 위해, 고려해야 할 핵심 요소가 들어갑니다. 이 핵심 요소는 산업 별로 정해져 있으나, 비즈니스 상황과 자사의 환경에 맞게 바꿀 수 있습니다. 마지막으로 큰 줄기에 매달린 잎 (갈색 네모난 부분)엔 비즈니스 목표를 달성함에 있어, 핵심 요소를 구성하는 데이터들을 작성해 줍니다. 실제 자사가 확보한 데이터를 작성해도 되고, 아직 확보되거나 구체적이지는 않지만, 핵심 요소에 꼭 필요한 데이터 까지 모두 작성해 줍니다. 서비스 산업에서는 주로 6P, 6가지 요소를 중점으로 어떤 데이터를 수집할지 파악합니다. 만약, 서비스업에서 6P라고 한다면, 각 요소들에서 얻거나 확인해야 할 데이터를 모두 도출해 내는 것입니다. (Process 과정 / Place 장소 / People 사람 / Plant 시설 / Policy 정책 / Product 제품 또는 서비스) 회의를 통해, 비즈니스 목적/목표를 달성하기 위해 고려 해야 할 주요 인자들을 작성해 보세요.

아래의 표는, 각 도메인 별로 특성 요인도를 어떤 식으로 정리하는 지 정리해 놓았습니다.

분야 / 목적주된 Fishbone 분류 체계주요 항목 예시
제조업4M / 5M / 6M / 8MMan, Machine, Material, Method, Measurement, Mother Nature 등
서비스업4S / 8PSurroundings, Suppliers, Systems, Skill, 또는 Product, Price 등
의료/병원6요소 모델People, Equipment, Materials, Process, Environment, Management
기계 정밀도5M + 환경 기반Man, Methods, Machines, Materials, Environment

이 특성 요인도는 비즈니스 상황과 현재 기업이 조사한 내용 및 배경을 바탕으로, 팀원들이 모여 작성하는 것이므로, 실제로 데이터 수집으로 연결 되지 않을 수 있습니다. 그럼 이것을 실제 데이터로 수집할 수 있는 단계까지 가져가야겠죠?

앞서 특성 요인도에서 작성된 모든 인자를 데이터로 모두 수집할 수 있다면 좋지만, 실제로는 그렇지 않을 때가 많습니다. 또한 특성 요인도 작성 단계에서 작성은 되었으나, 목적을 달성하기에는 그 중요도가 낮은 인자들도 존재합니다. 이 모든 데이터를 다 수집/구조화 할 수 없기에, 현장에서는 이를 평가하고 우선순위를 매겨 데이터를 관리합니다. 이를 “데이터 수집성 평가표 (또는 요인 중요도)”라고 부릅니다. 이 표는 특성 요인도에서 얻은 각 인자들을 수집 가능성(실제로 수집이 가능한 데이터인가?)과 분석 중요도(분석 및 서비스에 있어 얼마나 중요도를 갖는가?)라는 두 지표로 평가 합니다. 아래의 예시를 보시면 쉽게 이해할 수 있습니다.

특이한 점은, 위의 점수들이 1,3,9점 중에 하나로 매겨져 있는 것입니다. 이유는 데이터 수집/구조화에 있어, 변별하기 위함 입니다. 해당 점수는 각 팀원들이 점수를 모아 취합하여 최종 점수를 계산한 뒤, 최종 점수가 높은 데이터부터 수집 및 구조화를 수행합니다. 또는 기준 점수를 결정하여, 해당 점수 이상인 데이터에 대해 선별할 수 있습니다. 실제로는 굉장히 많은 인자들이 작성 되겠지요? 그러기에, 실제로 비즈니스 레포트나 발표용 자료엔 중요한 인자들만 위의 표처럼 간단하게 정리하고, 내부적으로는 프로젝트에 의해 고려된 모든 인자들에 대해 특성 요인도와 수집 평가표를 작성하여 팀원들 간 공유하면 됩니다.

앞서 수집 평가표에서 최종점수가 높은 인자들이 나열되고, 특정 점수 이상 넘는 인자에 대해 선별이 되면, 해당 인자들을 하나의 데이터 테이블 (또는 데이터 베이스)에 구조화 해야 합니다. 스키마, DB 설계서를 작성하기 전, 우리가 사용할 데이터를 정리하는 마지막 문서인데요. 바로 변수 정의서 입니다. Variable Definition Document라고 부르는 이 문서는 데이터를 인자(데이터 명), 타입(데이터 타입), 설명(데이터의 단위 또는 부가 설명)으로 나누어 정리합니다. 해당 자료를 사전에 작성하는 이유는, 팀원들 간 협업 또는 팀 간 협업에 있어, 서로가 생각하는 데이터를 같은 표현으로 소통하기 위함입니다. 가령 고객을 관리하는 CRM 시스템의 경우, “연령”이라는 데이터가 고객을 관리하는 직원의 연령인지, 고객의 연령인지 알 수 없습니다. 또 고객이 가입할 시점에서 연령인지, 현재 연도를 기준으로 계산된 연령인지도 헛깔릴 수 있습니다. 이는 DB 설계 단계 이전에 데이터를 명확히 정의하고, 데이터를 분석하여 표현하는 단계에서도 매우 치명적일 수 있습니다. 따라서 아래의 표와 같이 변수 정의서를 작성하여, 팀 간 협업에 있어 원활한 소통을 수행할 수 있습니다.

위의 표와 같이, 변수 정의서 차트가 여러 개로 되어 있는 것을 볼 수 있습니다. 여기서는 데이터 분석에 초점을 맞춰 테이블 별로 나눠 작성하였습니다. 실제 운영되고 있는 서비스에서 나온 데이터를 변수 정의서로 작성할 땐, DB설계서의 스키마 테이블 별로 정리할 수 있습니다. 데이터가 만약 수집된 상태라면, 수집된 데이터의 행의 개수와 열의 개수도 테이블 별로 작성해 놓아야 합니다. 이는 추후 데이터 분석을 수행함에 있어, 데이터의 행,열 수에 따라 분석 기법이 달라 질 수 있기 때문입니다. 타입에는 DB설계서와는 다르게, 연속형/범주형/날짜/ID(고유값) 형식으로 간단하게 정리되어 있습니다. 이를 더욱 구체화하여 컴퓨터에 표현될 형태로 정리할 때, 스키마에서는 더 디테일하게 자료형으로 나눠집니다. (Integer, float, string, object 등으로 표현) 이 단계에서는 간단하게 정리하는 이유는, 역시 팀 간, 부서 간 원할한 소통때문입니다. 같은 데이터라 하더라도 다른 서비스에서 데이터가 올라갈 때, 스키마 상 자료형은 달라질 수 있으므로, 변수 정의서에는 크게 숫자/문자/날짜/ID(고유값) 으로 분류를 해 놓습니다. 타입은 데이터 분석 기법을 선택할 때, 매우 중요하게 사용될 자료 이므로, 간단하게 작성하되, 절대 빼놓지는 말아야 합니다. 데이터 분석 관점에서는 기존에 정리해 놓은 변수 정의서에서 새로 “파생 변수”를 만들어 작성해 놓는 경우도 있습니다. 파생변수는 말 그대로, 기존에 수집 또는 구조화한 데이터에서 수식이나 데이터 간 연산을 통해 얻은 데이터로, 기존의 데이터로 부터, 디테일한 분석, 지표 관리 및 표현을 수행하기 위해 새로 만드는 변수라고 생각하면 쉽습니다. 파생 변수의 경우, 기존에 확보한 데이터를 기반으로 만든 변수이지만, 해당 변수를 사용하기 위한 목적에 따라 의미나 타입이 달라질 수 있으므로, 변수 정의서에 같이 작성합니다. 간단한 예로, “고객의 연령”이라는 데이터는 연속형(숫자) 데이터 입니다. 그러나 그래프로 표현을 하거나, 가설 검정을 수행할 때, 이 데이터를 연령대(문자)로 묶어 다른 형태의 분석 기법을 사용할 수 있습니다. 그럼 연령대(문자)는 기존에는 없었으나, 데이터 처리를 통해 얻을 수 있는 “파생변수”가 되는 것 입니다.

이제 변수 정의서까지 작성이 되었다면, DB설계서와 스키마 작성, 플로우 차트 구성, 데이터 분석 전 데이터를 구조화하는 작업을 문서화 해놓았다고 보시면 되겠습니다. 이제 변수 정의서를 바탕으로 데이터 분석가는 분석을, 엔지니어는 스키마 설계를, 서비스 개발자는 플로우 차트 구성 및 화면 기능 설계를 수행하면 됩니다. 같은 변수 정의서를 기반으로 설계를 수행하면, 부서 간 협업에 있어 큰 소요를 줄일 수 있고, 특정 시스템이나 설계를 보완 및 수정할 때도 매우 용이합니다. 보통 변수 정의서는 구글 시트나 ERD에서 사용하는 공용 프레드 시트에 정리를 합니다. 특성 요인도와 데이터 수집성 평가표는 변수 정의서를 작성하기 위한 준비 단계로 생각하시면 됩니다. 따라서 기획자들이 특성요인도 및 데이터 수집성 평가표를 작성하면, 분석가나 엔지니어가 수집 및 구조화가 가능한 데이터들을 정리하고, 이를 문서화하여 변수 정의서로 관리합니다. 관리하는 시스템의 규모가 클 경우, 변수 정의서는 여러 페이지, 수십, 수백장이 나올 수 있습니다. 추후에는 직급에 따라 접근 권한을 달리두어, 사내에서 관리하는 데이터에 변수 정의서와 특정 부서에서 운영하는 서비스의 변수 정의서를 분리하여 관리하기도 합니다. 이런 부분들을 고려하여, 앞서 주어진 비즈니스 시나리오를 기반으로 특성 요인도, 데이터 수집성 평가표, 변수 정의서를 작성하여 데이터를 관리해 봅시다.
